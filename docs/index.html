Turncraft · Prompt Eval Log

Conversation Guidelines — Rate Lock — Eval Log

Turncraft is a tool I built to design governed chatbot conversations: give it a user intent, and it maps the full scenario space. It then generates responses that follow a team's conversation guidelines and design system.

This prompt evaluation log documents what happened when I put the conversation guidelines under the same scrutiny the tool is meant to bring to a chatbot. It's built on a fixed set of test cases, a pre-defined scoring rubric, and repeat runs to tell a real pattern from a coincidence.

When a run fails, I trace why the guidelines allowed it, fix that specific gap, and check the fix with cold tests. I picked one scenario cluster, "locking in a preferential rate", on a trading bot in a fintech domain as the worked example. It's deliberately narrow, so the process could be documented all the way through instead of stretched thin across everything the tool can do.

Generation method

Responses were generated directly by reading the guidelines and scenario files and applying them, rather than by operating the deployed HTML artifact. This mirrors Turncraft's own mechanism — the tool's core function is "the model reads the loaded materials and generates the response" — so the same model applying the same materials in a different wrapper is a reasonable proxy, not a shortcut.

Runs per scenario: 3 (labeled A, B, C), generated independently, to test whether the same input produces consistent output — not just whether one output happens to follow the rules.

Known limitations of this pass

Documented rather than left implicit, since that's part of the practice being demonstrated here:

— No cross-check against the design-system materials (Material 3 / Ant Design X) for component-level accuracy; component names were checked against the guidelines' own descriptions only.

— No visibility into any prompt scaffolding inside the deployed artifact itself beyond the three materials files.

— Three runs is enough to detect gross inconsistency, not to establish a statistically stable failure rate.

Root cause grouping

A — Invented remedies aren't the problem; unresolved CTAs and dead ends are

S3S5S6

Turncraft is meant to suggest reasonable responses for states the guidelines don't explicitly cover — that's a feature, not a defect, and a tool that can only ever repeat pre-written material is worth less than one that can extend it sensibly. The actual failure mode isn't invention, it's leaving the user without a concrete next step, or letting a suggestion quietly take over the CTA slot instead of staying in the message copy.

ScenarioWhat's validWhat fails
S3Runs A and B — both CTAs resolve to viewing/managing existing locks. The tier-upgrade mention in Run B is fine as suggestive color, not as the CTA.Run C — no CTA at all. A dead end.
S5All three runs (view / cancel / complete) name a real action — none is wrong.None is complete alone. The fix is a component, not a CTA choice: show the pending lock as a card with "Cancel" and "Complete" actions in it.
S6Run A — "Notify me when available," the action a blocked user most likely wants first.Run B — silence, a dead end, unacceptable regardless of the invention principle. Run C — offers an alternative as a competing CTA instead of a suggestion in the message.
Fix (Trading flows subsection): "When a rate-lock-adjacent state has no explicit remedy defined, the bot may suggest context in its message, but the CTA must resolve to a concrete, always-available action — most often viewing or managing the user's existing state. Never end a turn without an actionable next step. Where a relevant lock or transaction already exists, render it as a card with its own actions (e.g. cancel, complete) rather than collapsing everything into one generic CTA."

B — A scenario definition bundles two distinct system states

S4

"Not verified or suspended" reads as one condition but requires two different governed responses. Each run picked a different assumption, and one run tried to hedge both in a single turn — which broke a separate rule (one focused action per turn). Confirmed as valid: the two states need to be differentiated, each resolving to one focused action.

Fix: split S4 into S4a (KYC incomplete → "Complete verification") and S4b (suspended/flagged → apply the Account Restrictions rule verbatim: never speculate why, provide the next verified step only, e.g. "Contact support") — one CTA per turn, never both. This is a scenario-authoring fix in fintech-scenarios.md, not the guidelines file — and it's the "missing scenarios" failure mode from the original Substack piece, showing up concretely.

C — "Lead with the number" has no fallback clause

S6S2

The rule assumes a number always exists to lead with. S6 has none (market's closed); S2's most relevant number is a shortfall, not a rate — the rule doesn't disambiguate which number applies when the obvious one is missing.

Fix: in the same Trading flows subsection, add one sentence — "If no rate is available, lead with the blocker instead of the rate."

D — Silence is not a valid resolution

S7

Guidelines don't currently say what happens when a user abandons a flow or lets it expire before confirming — so silence looked like a defensible option only because nothing ruled it out. On review, it isn't: every user-initiated action, including a cancellation, deserves acknowledgment. Run B ("Your rate lock offer has expired. Start a new lock anytime.") is the correct behavior; the silent runs are the failure.

Fix (Trading flows subsection): "If a user lets a rate lock offer expire or abandons before confirming, don't stay silent — acknowledge what happened and invite them to start again. Never leave a user action unaddressed."

Validation method — cold, independent runs

An earlier pass in this eval generated the v2 responses in the same session used to write the rules, by the same author, with full knowledge of exactly which behavior each new line was meant to produce. That's a source of bias no same-session pass can rule out on its own — convergence in that pass was partly expected by construction, not proof the rules read clearly to a model encountering them with no memory of writing them.

The results below replace that pass. They come from 3 independently spawned agents, each started fresh with no memory of this conversation, the eval history, the root causes, or what a "correct" answer was expected to look like. Each was given only the two source files — unified-conversation-design-principles-v2.md and the Rate Lock table in fintech-scenarios.md — and asked to generate the 8 scenario responses cold, with no other context. This is the strongest form of validation available without operating the deployed HTML artifact directly.

Root causes — resolution status

A (unavailable states — S3, S5, S6): resolved. The card-pattern default (S3, S5) holds independently and strongly. S6's open gap — no existing record to anchor a CTA to — is now covered by a new line: prioritize continuing the user's original goal over redirecting, and phrase any unconfirmed capability as a soft gesture rather than a guaranteed mechanism. Not yet re-validated cold.

B (bundled scenario — S4): resolved. S4a and S4b each resolve to one clean, single CTA across all 3 independent runs, with no hedging.

C (no fallback for "lead with the number"): resolved. S2 and S6 both correctly lead with the blocker, not the rate, across all runs.

D (silence on cancellation — S7): resolved. Every independent run closes the loop.

New, from cold validation: S1's scope ambiguity (full sequence vs. terminal state) wasn't visible in the same-session pass. Resolved with a scenario-authoring convention in fintech-scenarios.md: simulate every condition from its natural entry point, even when the label names an outcome. Not yet re-validated cold.

Still open: confirming S6 + S1's fixes hold under a second cold-validation pass. S1's CTA-wording drift (a separate, lower-severity item) remains deferred by design.

Applied

S6 — CTA direction, settled. The tension was between a redirect that's safe but abandons the user's original goal ("view other pairs") and an action that continues the goal but may not be a confirmed capability ("notify me when available"). Decision: prioritize continuing the user's original goal over redirecting to a different one, since a redirect reads as the bot giving up. Where the continuing action depends on an unconfirmed capability, phrase it as a soft, low-commitment gesture ("Want us to flag you when it's back?") rather than a guaranteed mechanism — honest either way, whether or not the feature exists. Only fall back to a plain redirect if no honest intent-continuing option exists, framed as a detour, not a replacement. Added as a new bullet in Trading flows, alongside — not replacing — the existing card-pattern line for scenarios with an existing record (S3, S5).

S1 — scope ambiguity, settled. The issue: S1 is the only scenario labeled with an outcome ("locked successfully") rather than a triggering condition, and nothing specified whether to simulate the full initiation-to-completion sequence or just the terminal state. Decision: scenarios are always simulated from their natural entry point through to resolution, even when the label names an outcome. Added as a methodology note in fintech-scenarios.md, directly above the Scenario Bank tables — a testing/authoring convention, not a change to how the bot behaves in production.

Review mechanism confirmed: the designer reviews generated responses and can suggest a tweak if a CTA isn't the intended one — this is the live feedback loop the guidelines are meant to support, not a one-time fix-and-forget.
